## 1.性能评估模型

### 核心目的

在做性能优化之前，首先要有一个衡量标准或模型，即“性能是什么”以及“如何评估”。这节旨在建立性能评估的视角与度量方式。

### 关键信息点与思考

- **性能指标**：可包括首屏时间、交互响应时间、帧率、内存占用、首字节时间 (TTFB)、总资源大小等。
- **性能预算 / 性能目标**：给每个页面或关键路径设置性能预算，比如最大资源大小、关键资源加载时间等。
- **关键路径**：分析从请求发起到页面完成渲染的路径中，哪些环节是“瓶颈”（网络、脚本阻塞、样式计算、布局、重绘等）。
- **量化 vs 定性**：性能评估既要有定性的用户感知，也要有定量的数据支持；比如页面流畅度 vs 帧率曲线、性能曲线图。

> 这个模型的意义在于：优化必须基于可度量、可追踪的目标，而不是盲目调整。

## 2 从 URL 到 图像（网络 + 渲染流程）

这一节是性能优化的“核心链路”分析：从资源请求到最终内容在屏幕上呈现，涉及两个大阶段 —— 网络层面 + 渲染层面。

### 2.1 网络

#### 关键因素 & 优化策略

- **请求数量与资源合并 / 拆分**：过多的小资源会增加 HTTP 请求开销；但又不能将所有资源打包过大。
- **资源优先级 & 延迟加载 / 懒加载**：关键资源优先加载，非关键图片 / 脚本延后加载。
- **缓存策略**：合理设置 HTTP 缓存、CDN 分发、版本号、缓存失效策略。
- **网络延迟与带宽**：人为缩短链路（CDN、边缘部署、减少跳数）、压缩资源（gzip / brotli / 压缩图片 / svg / WebP）
- **预加载 / 预连接 / DNS 预取**：使用 `<link rel="preload">`、`<link rel="prefetch">`、`dns-prefetch` 等机制提前准备资源。
- **资源压缩 / 精灵图 / 合图 / 雪碧图 / WebP / SVG** 等优化手段。

这部分主要从“URL 发起请求”到“资源加载完成”这一段链路来拆解网络性能的影响因素。 ([Cread](https://cread.jd.com/read/startRead.action?bookId=30746878&readType=1&utm_source=chatgpt.com))

### 2.2 渲染

在资源被加载之后，浏览器还要执行 CSS / JS / 布局 / 绘制 / 合成等操作，这里是“资源”变成“可视内容”的关键路径。

#### 关键因素 & 优化方向

- **样式计算 / CSS 选择器效率**：尽量使用简单、性能优的选择器，减少复杂的层级选择。
- **布局 (Layout / Reflow)、重排 / 重绘 (Repaint)**：避免频繁触发布局 / 重排；批量更改样式、使用 `transform` / `opacity` 优化动画。
- **合成 (Composite / GPU 加速)**：合理使用 GPU 图层，让某些元素单独成图层以减少重绘。
- **避免主线程阻塞**：避免长任务 (Long Task)、避免过大 JS 执行阻塞浏览器渲染。
- **渲染流水线优化**：Critical CSS（关键 CSS）、CSS 先行 / 延后加载、JS 延迟 / 异步执行。
- **分层 / 分块渲染**：对首屏内容优先渲染，其它部分延后。

这部分是经典 “Critical Rendering Path”（关键渲染路径）思想的体现：尽量压缩从 HTML → CSS → JS → 布局 → 绘制 → 合成为图像的时延。

## 3 内存管理

性能不仅是“响应 / 渲染速度”，还涉及长期运行、用户操作过程中资源使用是否健康。内存管理就是关注应用“常驻”状态的性能。

### 3.1 GC 算法

#### 重点内容

- 垃圾回收机制原理：标记-清除 (Mark-Sweep)、标记-整理 (Mark-Compact)、分代 GC 等。
- 不同 GC 算法的适用场景与性能特征：老年代、年轻代的切换、Minor GC / Major GC 的开销差异。
- 如何减少 GC 负担：减少临时对象、避免频繁创建销毁大量对象、对象池 / 复用等策略。

这部分让你理解：JavaScript 引擎中 GC 不是“黑盒”，其行为会对性能产生不可忽视的影响。

### 3.2 内存泄漏

#### 重点内容

- 常见内存泄漏类型：闭包泄漏、未移除的事件监听、定时器 / 回调未清理、DOM 引用未释放、全局变量、缓存对象持续增长等。
- 如何检测内存泄漏：使用浏览器开发者工具的 Heap Profiler、快照对比、分配剖析 (Allocation Profiling)、Timeline 追踪。
- 解决策略：及时销毁、取消绑定、弱引用 (WeakMap / WeakSet)、合理作用域控制、复用对象池等。

本节强调：长期运行的页面／单页应用如果忽略内存泄漏问题，会随着使用时间推移导致性能大幅下降甚至浏览器崩溃。

## 4 极限运算性能

这一节偏向“高计算 / 大数据 / 复杂运算”场景下的性能处理策略。

### 关键点与策略

- **Web Worker / Worker Threads**：将计算密集型任务放到后台线程，避免主线程卡顿。
- **分片 / 任务拆分**：将大任务拆成多个小任务，让浏览器有喘息机会保持主线程响应性。
- **GPU / WebGL / GPU 加速**：对于图形、动画、计算密集型任务，尝试使用 GPU 加速计算。
- **缓存 / 结果复用**：如果计算有重复子任务，可缓存其结果避免重复计算。
- **懒计算 / 按需计算**：只有在必要时才触发计算，不要过早或过多地执行复杂逻辑。

本节的目的在于：当应用遇到极限场景时（如大量数据处理、复杂动画、图形渲染等），应该有性能策略可用，而不是一味地“优化网络/渲染”那一路径。

---

## 5 总结

性能优化是一个全链路、持续的过程。不能只盯一个环节（如网络或 JS），而需从资源获取 → 渲染 → 运行时 → 长期稳定性 → 计算性能多个维度统筹考虑。
